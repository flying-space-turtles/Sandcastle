# Sandcastle CTF Infrastructure Scaffold

Sandcastle is a Docker-only scaffold for iterating on a local Attack & Defense
CTF environment. This stripped version keeps only the infrastructure layer:

- a deterministic Docker bridge network at `10.10.0.0/16`
- one SSH gateway container per team
- one vulnerable-application container slot per team
- scripts that generate, start, stop, and reset the topology

It intentionally does not include a vulnerable app, gameserver, checker,
scoreboard, dashboard, or scoring logic. Those pieces should be added back in
gradually as separate iterations.

## Layout

```text
.
├── docker-compose.yml          # generated by scripts/gen_compose.py
├── scripts/
│   ├── gen_compose.py          # parameterized compose generator
│   ├── setup.sh                # generate compose for N teams
│   ├── start.sh                # build SSH images + start infrastructure
│   ├── stop.sh                 # docker compose down
│   └── reset.sh                # docker compose down -v + start
├── services/
│   └── README.md               # contract for future vulnerable apps
├── teams/
│   └── ssh/
│       └── Dockerfile          # team SSH gateway image
└── docs/
    └── architecture.md         # infrastructure topology notes
```

## Requirements

- Docker 24+ with the `docker compose` plugin
- Python 3 for `scripts/gen_compose.py`

## Generate The Topology

```bash
./scripts/setup.sh        # default: 3 teams
./scripts/setup.sh 5      # generate 5 teams
```

The generator writes `docker-compose.yml`. Do not edit that file directly;
edit `scripts/gen_compose.py` and regenerate it.

Each generated team has:

| Container | Internal IP | Host access |
|---|---:|---|
| `team<N>-ssh` | `10.10.<N>.2` | SSH on host port `2200 + N` |
| `team<N>-vuln` | `10.10.<N>.3` | internal network only |

The SSH password is `team<N>pass`; for example, Team 3 uses host port `2203`
and password `team3pass`.

## Start And Stop

This scaffold does not bundle a vulnerable app image, so `VULN_IMAGE` is
required when starting:

```bash
VULN_IMAGE=<image-name-or-tag> ./scripts/start.sh
```

For example, after a future app is built locally as `sandcastle/example-vuln`:

```bash
VULN_IMAGE=sandcastle/example-vuln ./scripts/start.sh
```

Stop while keeping team data volumes:

```bash
./scripts/stop.sh
```

Reset volumes and restart:

```bash
VULN_IMAGE=<image-name-or-tag> ./scripts/reset.sh
```

## Future Vulnerable App Contract

Every `team<N>-vuln` service uses the image supplied through `VULN_IMAGE`.
The image is reused for every team and receives:

- `TEAM_ID=<N>`
- `TEAM_NAME=Team <N>`
- `SERVICE_PORT=${VULN_PORT:-8080}`
- a persistent volume mounted at `/app/data`
- network identity `team<N>-vuln` on `ctf-network`
- static IP `10.10.<N>.3`

The scaffold does not impose an HTTP API, flag format, checker behavior, or
scoring model yet.

## Security Note

The team SSH containers mount `/var/run/docker.sock` so Docker commands inside
the gateway talk to the host daemon. This is useful for local iteration but is
not a security boundary. Do not expose this scaffold to untrusted networks or
untrusted participants without replacing that model.
