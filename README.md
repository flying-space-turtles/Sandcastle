# Sandcastle CTF Infrastructure Scaffold

Sandcastle is a Docker-only scaffold for iterating on a local Attack & Defense
CTF environment. This stripped version keeps only the infrastructure layer:

- a deterministic Docker bridge network at `10.10.0.0/16`
- one SSH gateway container per team
- one vulnerable Linux machine container per team
- scripts that generate, start, stop, and reset the topology
- a template vulnerable service copied into each generated team workspace

It intentionally does not include a gameserver, checker, scoreboard, dashboard,
or scoring logic. Those pieces should be added back in gradually as separate
iterations.

## Layout

```text
.
├── docker-compose.yml          # generated by scripts/setup.sh
├── docker/
│   ├── firewall/
│   │   ├── Dockerfile          # firewall / traffic gateway image
│   │   └── rules.sh            # default iptables ruleset
│   ├── ssh/
│   │   └── Dockerfile          # reusable SSH gateway image
│   └── vuln/
│       └── Dockerfile          # reusable vulnerable-machine image
├── scripts/
│   ├── gen_compose.py          # compatibility wrapper for setup.sh
│   ├── setup.sh                # generate team directories + compose
│   ├── start.sh                # build SSH images + start infrastructure
│   ├── stop.sh                 # docker compose down
│   └── reset.sh                # docker compose down -v + start
├── services/
│   └── example-vuln/           # vulnerable service template
├── teams/
│   └── generated/              # ignored per-team app workspaces
├── visualizer/                 # React Flow Docker architecture visualizer
└── docs/
    └── architecture.md         # infrastructure topology notes
```

## Requirements

- Docker 24+ with the `docker compose` plugin
- Bash

## Generate The Topology

```bash
./scripts/setup.sh --teams 4      # generate 4 teams
./scripts/setup.sh                # default: 3 teams
```

The generator writes `docker-compose.yml` and creates
`teams/generated/team<N>/example-vuln` from `services/example-vuln`. Those
generated team workspaces are ignored by Git because they are runtime state:
teams patch them independently during a round. Do not edit `docker-compose.yml`
directly; re-run `scripts/setup.sh` instead.

The generated topology includes a central firewall container that all team
traffic routes through:

| Container | Internal IP | Host access |
|---|---:|---|
| `firewall` | `10.10.0.2` | (internal — default gateway) |
| `team<N>-ssh` | `10.10.<N>.2` | SSH on host port `2200 + N` |
| `team<N>-vuln` | `10.10.<N>.3` | SSH from `team<N>-ssh` |
| `team<N>-vuln-app` | `10.10.<N>.3` | shares `team<N>-vuln` networking |

The SSH username is `team<N>` and the password is `team<N>pass`; for example,
Team 3 uses host port `2203`, username `team3`, and password `team3pass`.

If you lower the team count, generated extra team directories are pruned. For
example, running `./scripts/setup.sh --teams 4` after a 6-team setup removes
generated `teams/generated/team5` and `teams/generated/team6` and rewrites
`docker-compose.yml`.

To seed teams from a different vulnerable-service template:

```bash
./scripts/setup.sh --teams 4 --template services/my-broken-service
```

Existing `teams/generated/team<N>/example-vuln` directories are preserved by
default so team patches are not overwritten. To intentionally replace them with
the template:

```bash
./scripts/setup.sh --teams 4 --template services/my-broken-service --overwrite-services
```

## Service Workspace Model

The source of truth for a challenge lives under `services/`. Setup copies that
template into ignored per-team workspaces under `teams/generated/`.

Each host-facing SSH gateway can reach its team's vulnerable Linux machine over
SSH. The vulnerable machine bind-mounts its generated app workspace at
`/home/team<N>/example-vuln` and has Docker CLI + Compose available through the
host Docker socket. That means a patch made inside `team<N>-vuln` changes the
exact source used for the next rebuild of that team's app container, without
affecting other teams.

This model is intentional: the committed repo keeps only templates and
infrastructure, while each round's mutable team code stays in ignored generated
directories. It avoids committing copied runtime state, keeps machine image
logic reusable, and still gives each team an independent app tree they can patch
and rebuild.

From the host, hop through the gateway into the vulnerable machine, then start
the bundled app:

```bash
ssh -p 2201 team1@localhost
ssh team1@team1-vuln
cd ~/example-vuln
vim app/app.py
docker compose up -d --build
curl http://team1-vuln:8080/health
```

The generated app Compose file recreates only that team's
`team<N>-vuln-app` container and uses a per-team data volume.

## Start And Stop

```bash
docker compose up --build
# or
./scripts/start.sh
```

Stop while keeping team data volumes:

```bash
./scripts/stop.sh
```

Reset volumes and restart:

```bash
./scripts/reset.sh
```

## Visualize The Docker Architecture

The `visualizer/` module is a standalone React + React Flow app that parses
Compose YAML and renders the infrastructure as an interactive network topology:

```bash
cd visualizer
npm install
npm run dev
```

It loads the repository root `docker-compose.yml` by default and includes a
YAML mode for pasted or uploaded Compose files.

## Future Vulnerable App Contract

Every `team<N>-vuln` machine mounts `teams/generated/team<N>/example-vuln`,
which starts as a copy of `services/example-vuln`. Running `docker compose up`
inside that directory starts `team<N>-vuln-app`, which receives:

- `TEAM_ID=<N>`
- `TEAM_NAME=Team <N>`
- `SERVICE_PORT=8080`
- `SECRET_KEY=sandcastle-team<N>-dev-secret`
- a persistent volume named `sandcastle_team<N>-data` mounted at `/app/data`
- shared network identity with `team<N>-vuln` on `ctf-network`
- static IP `10.10.<N>.3`

The scaffold does not impose an HTTP API, flag format, checker behavior, or
scoring model yet.

## Firewall

All inter-container traffic flows through the firewall container at `10.10.0.2`.
Each team container's entrypoint overrides the default route to point at the
firewall when the `GATEWAY_OVERRIDE` environment variable is set.

The firewall runs `docker/firewall/rules.sh` at startup, which applies iptables
rules. The default ruleset is permissive (FORWARD ACCEPT) and logs every new
connection. To customise the rules, edit `docker/firewall/rules.sh` and
rebuild:

```bash
docker compose up --build firewall
```

Example: block team 1 from attacking team 2's service:

```bash
# Add to docker/firewall/rules.sh before the exec line:
iptables -A FORWARD -s 10.10.1.0/24 -d 10.10.2.3 -p tcp --dport 8080 -j DROP
```

## Security Note

The vulnerable machine containers mount `/var/run/docker.sock` so Docker
commands inside `team<N>-vuln` talk to the host daemon. This is useful for local
iteration but is not a security boundary. Do not expose this scaffold to
untrusted networks or untrusted participants without replacing that model.
